# Offline Demo Package Spec

## Problem

当前仓库的主链路依赖 Next.js BFF 和 `/api/*` 网络请求，不满足互动空间 Demo 的上传约束：

- 包体需要控制在 8MB 以内
- 运行环境不支持网络通信
- 扫码展示需要一个稳定、可离线运行的独立 Demo

如果继续直接使用当前主站构建结果，Demo 在互动空间中会因为请求失败而无法完成核心流程。

## Goal

新增一个独立的离线 `demo/` 目录，通过本地静态页面和本地 mock 数据复刻四页核心流程，并提供可直接上传的构建产物目录。

## Non Goals

- 不替换当前完整版 Next.js 主站
- 不修改现有 `/api/generate`、`/api/feedback`、`/api/track` 后端链路
- 不把离线 Demo 接入真实模型或任何远程服务
- 不新增登录、存储同步、复杂埋点或数据库

## User Stories

- 作为活动参与者，我希望扫码后看到一个无需联网也能操作的完整 Demo，这样互动空间中不会因为接口失败而中断体验。
- 作为项目维护者，我希望通过单独命令生成离线 Demo 目录，这样可以把互动空间用的 Demo 和现场展示的完整版分开交付。
- 作为演示用户，我希望在 Demo 中看到覆盖六种风格的本地示例结果，这样可以真实感受到产品能力而不是空壳页面。

## Acceptance Criteria

- [ ] 仓库新增独立 `demo/` 目录，包含首页、输入页、语气页、结果页四个可直接访问的静态页面。
- [ ] Demo 页面运行时不发起任何网络请求，生成结果、反馈提示和页面数据均来自本地文件。
- [ ] Demo 内置 20 条 mock 数据，覆盖 `delay`、`refuse`、`boundary`、`followup`、`decode`、`sarcasm` 六种风格。
- [ ] 根目录提供明确的离线构建命令，执行后输出独立的可上传目录 `demo/dist/`。
- [ ] 主站现有 Next.js 页面、BFF API、模型链路和默认构建命令保持不变。
- [ ] 文档补充离线 Demo 的用途、构建方式和与完整版的区别。

## Edge Cases

- 输入页为空或不足 2 个字时，不能继续进入结果生成，应提示先补充原话。
- 用户未完成场景选择或没有填写原话时，语气页和结果页需要给出回跳提示。
- 当输入内容无法精确匹配某条 mock 数据时，结果页应按场景 + 风格优先匹配，并回退到该风格默认样例。
- Demo 需要支持本地文件方式访问或静态托管访问，页面资源路径必须使用相对路径。
- 构建脚本重复执行时，应先清理旧的 `demo/dist/`，避免残留过期文件。
